问题 1: 断点续传失效
================================================================================

【问题描述】
PDF 翻译不支持真正的断点续传，当翻译中断后只能从第1页重新开始。

【代码位置】
PdfDocumentTranslator.cs 第27-30行:
    if (context.ResumeUnitIndex > 0)
    {
        Log($"PDF 暂不支持从第 {context.ResumeUnitIndex + 1} 页继续写入，已自动从头开始重新生成。");
    }

【根本原因】
1. PDF文件格式是追加式的，PdfSharp库不支持修改已写入的页面
2. 输出PDF是逐步构建的，中断时已有部分页面写入，但无法恢复中间状态

【影响】
- 大文件（数百页）翻译中断后需要完全重做
- 浪费时间和API调用费用
- 用户体验差

【解决方案】
方案A: 分阶段输出（推荐）
1. 将PDF按页拆分为临时单页PDF文件
2. 每页翻译完成后立即保存为独立文件
3. 全部完成后合并所有单页PDF
4. 检查点记录已完成的页码，支持跳过

代码实现思路:
```csharp
public override async Task TranslateAsync(TranslationJobContext context)
{
    var tempDir = Path.Combine(Path.GetTempPath(), $"pdf_translation_{Guid.NewGuid()}");
    Directory.CreateDirectory(tempDir);

    try
    {
        for (var pageIndex = 0; pageIndex < pageCount; pageIndex++)
        {
            var tempPagePath = Path.Combine(tempDir, $"page_{pageIndex:D4}.pdf");

            // 检查是否已翻译
            if (pageIndex < context.ResumeUnitIndex && File.Exists(tempPagePath))
            {
                continue;
            }

            // 翻译单页并保存
            await TranslateSinglePageAsync(context, pageIndex, tempPagePath);
            await context.SaveCheckpointAsync(pageIndex + 1, 0, $"已完成第{pageIndex + 1}页");
        }

        // 合并所有页面
        MergePdfPages(tempDir, outputPath);
    }
    finally
    {
        Directory.Delete(tempDir, true);
    }
}
```

方案B: 使用支持增量写入的PDF库
- 考虑使用 iTextSharp 或 PdfBox 等支持增量更新的库
- 但会增加依赖复杂度

问题 2: 内存使用过高
================================================================================

【问题描述】
同时打开三个PDF文档对象，大文件可能导致内存溢出。

【代码位置】
PdfDocumentTranslator.cs 第32-34行:
    using var inputPdf = PdfReader.Open(context.Item.SourcePath, PdfDocumentOpenMode.Import);
    using var pig = UglyToad.PdfPig.PdfDocument.Open(context.Item.SourcePath);
    using var outputPdf = new PdfSharp.Pdf.PdfDocument();

【根本原因】
1. PdfSharp 导入模式需要保持输入文档打开
2. PdfPig 需要独立打开文档进行文本提取
3. 输出文档需要缓存所有页面直到保存

【影响】
- 大文件（数百页）可能导致 OutOfMemoryException
- 系统性能下降

【解决方案】
方案A: 流式处理（推荐）
1. 逐页读取、翻译、写入
2. 每页处理完成后释放资源
3. 使用 PdfSharp 的流式写入模式

代码实现:
```csharp
public override async Task TranslateAsync(TranslationJobContext context)
{
    // 只获取页数信息，不加载全部
    var pageCount = GetPageCount(context.Item.SourcePath);

    // 使用流式写入
    using var outputStream = File.Create(outputPath);
    using var writer = new PdfWriter(outputStream);

    for (var pageIndex = 0; pageIndex < pageCount; pageIndex++)
    {
        // 每页独立打开和关闭
        using var pageInput = OpenSinglePage(context.Item.SourcePath, pageIndex);
        using var pigPage = ExtractTextFromPage(context.Item.SourcePath, pageIndex);

        var translatedPage = await TranslatePageAsync(pageInput, pigPage);
        writer.AddPage(translatedPage);

        // 强制垃圾回收
        if (pageIndex % 10 == 0)
        {
            GC.Collect();
        }
    }
}
```

方案B: 分页缓存策略
- 将大文件分成多个批次处理
- 每批处理N页后保存中间结果
- 最后合并

问题 3: OCR坐标计算错误
================================================================================

【问题描述】
OCR渲染缩放后的坐标计算不准确，导致译文位置偏移。

【代码位置】
OcrService.cs 第38-79行:
    var renderScale = Math.Clamp(settings.RenderScale, MinRenderScale, MaxRenderScale);
    using var pageReader = docReader.GetPageReader(pageIndex);
    var width = pageReader.GetPageWidth();  // 返回缩放后的宽度
    var height = pageReader.GetPageHeight(); // 返回缩放后的高度
    // ...
    words.Add(new OcrWord(
        text.Trim(),
        rect.X1 / renderScale,  // 坐标已缩放
        rect.Y1 / renderScale,
        rect.X2 / renderScale,
        rect.Y2 / renderScale,
        confidence));

【根本原因】
1. Docnet 返回的宽高是缩放后的值
2. 但 OCR 识别的坐标是基于渲染图片的
3. 坐标转换时除以 renderScale，但宽高未做对应处理

【影响】
- OCR识别的文本块位置偏移
- 译文覆盖位置不正确
- 可能出现重叠或错位

【解决方案】
```csharp
// 修正后的代码
var renderScale = Math.Clamp(settings.RenderScale, MinRenderScale, MaxRenderScale);
using var pageReader = docReader.GetPageReader(pageIndex);

// 获取原始尺寸
var originalWidth = pageReader.GetPageWidth() / renderScale;
var originalHeight = pageReader.GetPageHeight() / renderScale;

// 或者使用未缩放的获取方式
var pageDimensions = pageReader.GetPageDimensions();
var originalWidth = pageDimensions.Width;
var originalHeight = pageDimensions.Height;

// OCR坐标转换
words.Add(new OcrWord(
    text.Trim(),
    rect.X1 / renderScale,
    rect.Y1 / renderScale,
    rect.X2 / renderScale,
    rect.Y2 / renderScale,
    confidence));
```

问题 4: 并发控制配置混淆
================================================================================

【问题描述】
使用 MaxParallelDocuments 控制文本块级并发，语义混淆。

【代码位置】
DocumentTranslatorBase.cs 第120-121行:
    protected static int GetBlockTranslationConcurrency(AppSettings settings) =>
        Math.Clamp(settings.Translation.MaxParallelDocuments, 1, DefaultBlockTranslationConcurrency);

【根本原因】
配置项名称是"文档并发"，但实际用于"文本块并发"。

【影响】
- 用户配置与行为不符
- 难以调优性能

【解决方案】
```csharp
// 添加独立的配置项
public class TranslationSettings
{
    public int MaxParallelDocuments { get; set; } = 3;  // 文档级并发
    public int MaxParallelBlocks { get; set; } = 3;     // 块级并发（新增）
}

// 修改获取方法
protected static int GetBlockTranslationConcurrency(AppSettings settings) =>
    Math.Clamp(settings.Translation.MaxParallelBlocks, 1, DefaultBlockTranslationConcurrency);
```

问题 5: 公式检测过于激进
================================================================================

【问题描述】
普通文本中的数学表达式会被误判为公式而跳过翻译。

【代码位置】
PdfDocumentTranslator.cs 第1259-1276行:
    private static bool IsFormulaLikeBlock(PdfTextBlock block)
    {
        var hasEquationOperators = compact.Any(ch => "=+-*/^_∑∫∇√≈≤≥∈∀∂".Contains(ch));
        // ...
        return (hasEquationOperators && hasManyMathMarkers) || hasVariablePattern || mostlyShortSymbols;
    }

【根本原因】
检测规则过于宽泛，包含常见标点如等号、加减号。

【影响】
- "成本=价格*数量" 等业务文本被跳过
- 包含数字的列表被误判

【解决方案】
```csharp
private static bool IsFormulaLikeBlock(PdfTextBlock block)
{
    var text = block.Text.Trim();
    if (string.IsNullOrWhiteSpace(text))
        return false;

    var compact = text.Replace(" ", "").Replace("\n", "");

    // 提高阈值：公式通常很短
    if (compact.Length > 100)
        return false;

    // 严格检测：必须包含数学专用符号
    var mathSpecificSymbols = "∑∫∇√∂∈∀∃∄∅∆∏∐∑−∕∖∗∘∙√∛∜∞∟∠∡∢∣∤∥∦∧∨∩∪∫∬∭∮∯∰∱∲∳∴∵∶∷∸∹∺∻∼∽∾∿≀≁≂≃≄≅≆≇≈≉≊≋≌≍≎≏≐≑≒≓≔≕≖≗≘≙≚≛≜≝≞≟≠≡≢≣≤≥≦≧≨≩≪≫≬≭≮≯≰≱≲≳≴≵≶≷≸≹≺≻≼≽≾≿⊀⊁⊂⊃⊄⊅⊆⊇⊈⊉⊊⊋⊌⊍⊎⊏⊐⊑⊒⊓⊔⊕⊖⊗⊘⊙⊚⊛⊜⊝⊞⊟⊠⊡⊢⊣⊤⊥⊦⊧⊨⊩⊪⊫⊬⊭⊮⊯⊰⊱⊲⊳⊴⊵⊶⊷⊸⊹⊺⊻⊼⊽⊾⊿⋀⋁⋂⋃⋄⋅⋆⋇⋈⋉⋊⋋⋌⋍⋎⋏⋐⋑⋒⋓⋔⋕⋖⋗⋘⋙⋚⋛⋜⋝⋞⋟⋠⋡⋢⋣⋤⋥⋦⋧⋨⋩⋪⋫⋬⋭⋮⋯⋰⋱⋲⋳⋴⋵⋶⋷⋸⋹⋺⋻⋼⋽⋾⋿";
    var hasMathSpecificSymbol = compact.Any(ch => mathSpecificSymbols.Contains(ch));

    // 变量模式检测（更严格）
    var hasVariablePattern = Regex.IsMatch(text, @"\b[a-zA-Z]_{?\d+}?\b") ||  // x_1, x_{12}
                             Regex.IsMatch(text, @"\b[a-zA-Z]\^[a-zA-Z0-9]\b") ||  // x^2
                             Regex.IsMatch(text, @"\\\w+\{");  // LaTeX 命令

    // 公式通常包含括号配对
    var hasBalancedDelimiters =
        (text.Count(c => c == '(') == text.Count(c => c == ')')) &&
        (text.Count(c => c == '[') == text.Count(c => c == ']')) &&
        text.Contains('(');

    // 综合判断
    return (hasMathSpecificSymbol && hasBalancedDelimiters) ||
           (hasVariablePattern && compact.Length < 50);
}
```

问题 6: 字体创建可能崩溃
================================================================================

【问题描述】
字体创建失败时没有异常处理，可能导致程序崩溃。

【代码位置】
PdfDocumentTranslator.cs 第551-570行:
    for (var size = preferredSize; size >= 6; size -= 0.5)
    {
        var font = CreateFont(family, size, isBold, isItalic);  // 可能抛出异常
        // ...
    }

【根本原因】
XFont 构造函数在字体不存在时会抛出异常。

【解决方案】
```csharp
private static (XFont Font, IReadOnlyList<string> Lines) FitWrappedText(...)
{
    text ??= string.Empty;
    family = string.IsNullOrWhiteSpace(family) ? PdfSharpFontResolver.DefaultFontFamily : family;

    for (var size = preferredSize; size >= 6; size -= 0.5)
    {
        try
        {
            var font = CreateFont(family, size, isBold, isItalic);
            var lines = WrapText(graphics, text, font, maxWidth);
            var lineHeight = GetLineHeight(font, lineHeightRatio);
            var totalHeight = lines.Count * lineHeight;
            if (totalHeight <= maxHeight)
            {
                return (font, lines);
            }
        }
        catch (Exception ex)
        {
            // 记录日志但继续尝试更小字号
            Debug.WriteLine($"字体创建失败: {family}, size={size}, error={ex.Message}");
        }
    }

    // 最终回退到默认字体
    var fallbackFont = CreateFont(PdfSharpFontResolver.DefaultFontFamily, 6, isBold, isItalic);
    return (fallbackFont, WrapText(graphics, text, fallbackFont, maxWidth));
}
```

问题 7: 进度报告不准确
================================================================================

【问题描述】
进度基于页数而非文本量，单页大文件进度跳跃。

【代码位置】
PdfDocumentTranslator.cs 第106-107行:
    var progress = (int)Math.Round((pageIndex + 1) * 100d / Math.Max(1, inputPdf.PageCount));

【解决方案】
```csharp
// 基于文本量的进度计算
private double _totalTextLength;
private double _translatedTextLength;

public override async Task TranslateAsync(TranslationJobContext context)
{
    // 预扫描：计算总文本量
    _totalTextLength = EstimateTotalTextLength(context.Item.SourcePath);
    _translatedTextLength = 0;

    for (var pageIndex = 0; pageIndex < pageCount; pageIndex++)
    {
        var pageTextLength = await TranslateSinglePageAsync(...);
        _translatedTextLength += pageTextLength;

        var progress = (int)Math.Round(_translatedTextLength * 100d / Math.Max(1, _totalTextLength));
        await context.ReportProgressAsync(progress, $"已翻译 {progress}%");
    }
}

private double EstimateTotalTextLength(string pdfPath)
{
    using var pig = UglyToad.PdfPig.PdfDocument.Open(pdfPath);
    var totalLength = 0;
    foreach (var page in pig.GetPages())
    {
        totalLength += page.GetWords().Sum(w => w.Text.Length);
    }
    return totalLength;
}
```

public async Task ExportAsync(string sourcePath, string outputDirectory,
    IAsyncEnumerable<BilingualSegment> segments, CancellationToken cancellationToken)
{
    using var document = WordprocessingDocument.Create(bilingualPath, WordprocessingDocumentType.Document);
    var mainPart = document.AddMainDocumentPart();
    mainPart.Document = new Document(new Body());
    var body = mainPart.Document.Body!;

    // 创建表格
    var table = new Table();
    table.AppendChild(CreateTableProperties());
    table.Append(CreateRow("上下文", "原文", "译文", isHeader: true));

            mainPart.Document.Save();
        }
    }

    mainPart.Document.Save();
}

问题 9: 暂停控制器竞态条件
================================================================================

【问题描述】
快速暂停/恢复可能导致状态不一致。

【代码位置】
PauseController.cs 第12-32行

【解决方案】
```csharp
public sealed class PauseController
{
    private readonly object _lock = new();
    private TaskCompletionSource<bool> _resumeSignal;
    private bool _isPaused;

    public void Pause()
    {
        lock (_lock)
        {
            if (_isPaused) return;

            _isPaused = true;
            _resumeSignal = new TaskCompletionSource<bool>(TaskCreationOptions.RunContinuationsAsynchronously);
        }
    }

    public void Resume()
    {
        lock (_lock)
        {
            if (!_isPaused) return;

            _isPaused = false;
            _resumeSignal.TrySetResult(true);
        }
    }

    public Task WaitIfPausedAsync(CancellationToken cancellationToken)
    {
        TaskCompletionSource<bool> signal;

        lock (_lock)
        {
            if (!_isPaused) return Task.CompletedTask;
            signal = _resumeSignal;
        }

        return signal.Task.WaitAsync(cancellationToken);
    }
}
```

问题 12: 颜色转换精度损失
================================================================================

【问题描述】
使用简化的CMYK转换公式，颜色可能失真。

【代码位置】
PdfDocumentTranslator.cs 第1597-1606行

【解决方案】
```csharp
private static XColor ConvertCmykColor(CMYKColor cmyk)
{
    // 使用标准的CMYK到RGB转换
    // R = 255 * (1 - C) * (1 - K)
    // G = 255 * (1 - M) * (1 - K)
    // B = 255 * (1 - Y) * (1 - K)

    var r = (int)(255 * (1 - Math.Min(1, cmyk.C)) * (1 - Math.Min(1, cmyk.K)));
    var g = (int)(255 * (1 - Math.Min(1, cmyk.M)) * (1 - Math.Min(1, cmyk.K)));
    var b = (int)(255 * (1 - Math.Min(1, cmyk.Y)) * (1 - Math.Min(1, cmyk.K)));

    return XColor.FromArgb(255,
        (byte)Math.Clamp(r, 0, 255),
        (byte)Math.Clamp(g, 0, 255),
        (byte)Math.Clamp(b, 0, 255));
}
```

问题 13: 格式一致性问题汇总
================================================================================

【当前保证的格式】
✅ 页面尺寸和边距 - 直接复制原页面
✅ 图片/图形位置 - 原样保留
✅ 文本大致位置 - 基于PdfPig提取的坐标
⚠️ 阅读顺序 - 多列布局检测算法

【可能变化的格式】
⚠️ 字体外观 - 映射到系统字体
⚠️ 背景 - 白色覆盖层可能遮挡复杂背景
⚠️ 精细排版 - 字间距、行距微调
❌ 公式 - 直接跳过不翻译

【改进方案】

1. 透明覆盖层替代白色背景
```csharp
// 使用页面背景色采样而非纯白
var backgroundColor = SampleBackgroundColor(importedPage, block.Rect);
graphics.DrawRectangle(new XSolidBrush(backgroundColor), rect);
```

2. 字体嵌入支持
```csharp
// 检查原PDF是否嵌入字体，尝试提取使用
var embeddedFont = TryExtractEmbeddedFont(page, block.FontFamily);
if (embeddedFont != null)
{
    // 使用嵌入字体
}
```

3. 公式识别与渲染
```csharp
// 使用MathPix或本地LaTeX识别公式
var latex = await FormulaRecognizer.RecognizeAsync(block.Image);
var formulaImage = await LatexRenderer.RenderAsync(latex);
// 将公式图片绘制到PDF
```

4. 表格结构保留
```csharp
// 检测表格结构而非作为普通文本
var tableStructure = TableDetector.Detect(blocks);
foreach (var cell in tableStructure.Cells)
{
    // 保持单元格边界和对齐
    DrawTranslatedCell(graphics, cell);
}
```
